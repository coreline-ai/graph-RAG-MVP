## 1. 제품 개요

### 1.1 제품명
**Simple-Graph-RAG** — Neo4j + PostgreSQL/pgvector 기반 한국어 GraphRAG 질의응답 시스템

### 1.2 제품 목적
조직 내 한국어 채팅 로그와 대화 기록을 구조화된 그래프로 변환하고,
`bge-m3` 기반 한국어 임베딩으로 청크 벡터를 생성해
**PostgreSQL + pgvector**에 저장한 뒤, Neo4j GraphRAG 검색과 결합해
더 정확한 질의응답을 제공한다.

1차 구현 우선순위는 **Neo4j GraphRAG 중심 검색 흐름**이며,
핵심 검색 순서는 `날짜/권한/채널 필터 -> 벡터 검색 -> 그래프 확장`이다.

답변 생성 계층은 애플리케이션이 OpenAI API 키를 직접 저장하지 않고,
**Codex CLI에 ChatGPT 계정으로 인증된 세션을 사용하는 로컬 OAuth Proxy 어댑터**
방식으로 운영한다.

기존 `chroma_db` 디렉터리 기반 ChromaDB 저장 방식은 사용하지 않는다.

### 1.3 핵심 가치
| 가치 | 설명 |
|------|------|
| **Graph-first Retrieval** | 단순 벡터 검색이 아니라 엔티티, 관계, 시간축을 함께 활용 |
| **한국어 대화 최적화** | KSS 기반 한국어 청킹과 `bge-m3` 계열 임베딩 적용 |
| **정확한 근거 제시** | 답변마다 관련 청크, 그래프 이웃, 메타데이터 출처 반환 |
| **비밀정보 최소화** | 앱 레벨 OpenAI API 키 없이 Codex CLI 인증 세션을 프록시로 사용 |
| **운영 친화적 벡터 저장소** | 기본 벡터 저장소를 PostgreSQL/pgvector로 통일 |
| **확장 가능한 지식 구조** | 문서 RAG에서 GraphRAG로 확장 가능한 데이터 모델 |

### 1.4 대상 사용자
- 사내 메신저 로그에서 맥락과 관계까지 함께 검색해야 하는 팀 리더
- 특정 사용자, 채팅방, 이슈, 기간의 흐름을 빠르게 파악하려는 운영자
- 한국어 GraphRAG 파이프라인을 설계/구현하려는 개발자

### 1.5 제품 원칙
- 한국어 로그의 문맥 단위를 깨지 않는 청킹을 우선한다.
- 한국어 청킹 기본 구현은 `KSS`를 사용한다.
- 검색은 항상 `벡터 + 그래프 + 메타데이터` 관점에서 설계한다.
- 검색 실행 순서는 `날짜/권한/채널 필터 -> 벡터 검색 -> 그래프 확장`으로 고정한다.
- 답변 품질보다 먼저 근거 추적성과 재현성을 확보한다.
- 외부 모델 사용 시에도 앱 내부에는 장기 API 비밀값을 저장하지 않는다.
- 벡터 저장소 기본값은 PostgreSQL + pgvector이며 ChromaDB는 사용하지 않는다.

## 4. 데이터 형식 명세

### 4.1 입력 데이터
```text
[2024-02-21, 14:30:00, 백엔드개발, 서버 배포 3차 완료했습니다, 김민수]
```

| 필드 | 타입 | 형식 | 예시 |
|------|------|------|------|
| 날짜 | 문자열 | YYYY-MM-DD | 2024-02-21 |
| 시간 | 문자열 | HH:MM:SS | 14:30:00 |
| 채팅방이름 | 문자열 | 자유 | 백엔드개발 |
| 입력내용 | 문자열 | 자유 | 서버 배포 3차 완료했습니다 |
| 사용자 | 문자열 | 한글 이름 | 김민수 |

### 4.2 청크 데이터
```python
{
    "chunk_id": "doc_001_chunk_042",
    "chunk_text": "백엔드개발 김민수: 서버 배포 3차 완료했습니다",
    "original": "[2024-02-21, 14:30:00, 백엔드개발, 서버 배포 3차 완료했습니다, 김민수]",
    "metadata": {
        "channel": "백엔드개발",
        "user": "김민수",
        "date": "2024-02-21",
        "time": "14:30:00",
        "document_id": "doc_001",
        "access_scope": "team:backend"
    }
}
```

### 4.3 그래프 표현 예시
```python
{
    "nodes": [
        {"label": "Chunk", "id": "doc_001_chunk_042"},
        {"label": "User", "name": "김민수"},
        {"label": "Channel", "name": "백엔드개발"},
        {"label": "Entity", "name": "서버 배포"}
    ],
    "relationships": [
        {"type": "SENT_BY", "from": "doc_001_chunk_042", "to": "김민수"},
        {"type": "IN_CHANNEL", "from": "doc_001_chunk_042", "to": "백엔드개발"},
        {"type": "MENTIONS", "from": "doc_001_chunk_042", "to": "서버 배포"}
    ]
}
```

## 5. API 요구사항

### 5.1 POST /documents
```json
{
    "filename": "chat_logs.txt",
    "content": "[2024-02-21, 14:30:00, 백엔드개발, 서버 배포 3차 완료했습니다, 김민수]"
}
```

### 5.2 POST /query
```json
{
    "question": "2024년 3월 개발팀에서 배포와 관련된 흐름을 요약해줘",
    "top_k": 10
}
```

### 5.3 POST /query 응답
```json
{
    "question": "2024년 3월 개발팀에서 배포와 관련된 흐름을 요약해줘",
    "answer": "2024년 3월 개발팀에서는 ...",
    "retrieval_strategy": "graph_hybrid",
    "sources": [
        {
            "chunk_id": "doc_001_chunk_042",
            "content": "[2024-03-05, 10:20:00, 개발팀, 서버 배포 2차 완료했습니다, 박서준]",
            "score": 0.91,
            "graph_path": ["개발팀", "서버 배포", "박서준"]
        }
    ]
}
```

### 5.4 GET /health
```json
{
    "status": "ok",
    "neo4j": "ok",
    "postgres": "ok",
    "embedding": "ok",
    "codex_proxy": "ok"
}
```

## 6. 질의 유형 지원 매트릭스

| 질의 유형 | 예시 | 분석 결과 | 검색 전략 |
|----------|------|----------|----------|
| 내용 검색 | "서버 배포 관련 대화" | keyword | no_filter -> pgvector -> graph_expand |
| 날짜 검색 | "2024-02-21 일어난 일" | date | filter -> pgvector -> graph_expand |
| 날짜 범위 | "2024년 3월 대화" | date_from~to | filter -> pgvector -> graph_expand |
| 채널 검색 | "AI연구팀 대화" | channel | filter -> pgvector -> graph_expand |
| 사용자 검색 | "김민수가 뭐 했어?" | user | filter -> pgvector -> graph_expand |
| 권한 범위 검색 | "내 권한으로 볼 수 있는 배포 기록" | access_scope | filter -> pgvector -> graph_expand |
| 관계형 질문 | "김민수와 박서준이 같이 언급된 이슈는?" | relationship | graph_path |
| 타임라인 | "배포 이슈가 어떻게 진행됐어?" | timeline | temporal graph |
| 목록/나열 | "AI연구팀 팀원 이름" | list | graph aggregate |
| 통계/집계 | "가장 활발한 채널" | aggregate | Cypher aggregate |
| 요약 | "어제 무슨 일이 있었나" | summary | graph_hybrid + LLM |

## 7. 제약사항 및 리스크

| ID | 항목 | 영향 | 대응 방향 |
|----|------|------|----------|
| C-01 | 한국어 엔티티 추출 품질 편차 | 관계 품질 저하 | 규칙 기반 + 모델 기반 혼합 |
| C-02 | Neo4j + PostgreSQL 이중 저장소 운영 복잡도 | 초기 구현 비용 증가 | ID 동기화와 적재 파이프라인 표준화 |

| C-06 | KSS 분절 결과가 로그 턴 의미와 어긋날 수 있음 | 청킹 품질 저하 | 턴 병합 규칙과 후처리 |
| C-07 | 한국어 동음이의어/이름 오탐 | 검색 정확도 저하 | 사전, 블랙리스트, 후처리 |
| C-08 | 너무 작은 청크는 관계 정보가 부족함 | GraphRAG 성능 저하 | 턴 병합 규칙 도입 |
